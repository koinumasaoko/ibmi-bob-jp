# RPG III 画面開発規約

## 概要
このルールは、IBM i (AS/400) 上でのRPG III対話型プログラム開発に関する規約をまとめたものです。
画面ファイル（DDS）とRPGプログラムの連携、サブファイルの使用方法などを定義します。

## 1. ファイル命名規則

### 1.1 RPG IIIの制限
- **F仕様書のファイル名**: 最大8文字まで
- **例**: `IPH210L2`（8文字）は可、`IPH210SL2`（9文字）は不可

### 1.2 命名パターン
- 対話型プログラム: `iph###.rpg`
- 画面ファイル: `iph###s.dspf` または `iph###l#.dspf`（Lab番号付き）
- バッチプログラム: `bch###.rpg`

## 2. H仕様書（ヘッダー仕様）

### 2.1 標準形式
```rpg
     H*****************************************************************
     H* プログラム名 * プログラムの説明
     H*****************************************************************
     H            Y
```

### 2.2 日付編集コード
- `H            Y`: 日付を `YY/MM/DD` 形式で編集
- 対話型プログラムでは通常 `Y` を指定

## 3. F仕様書（ファイル定義）

### 3.1 画面ファイルの定義
```rpg
     FIPH210L2CF  E                    WORKSTN
     F                                        RRN01 KSFILE PANEL80
```

**重要なポイント**:
- ファイル名の後にスペースなし: `FIPH210L2CF`
- `CF`: 複合ファイル（Combined File）
- スペース2つ: `CF  E`
- `RRN01`: 相対レコード番号（サブファイル用）
- `KSFILE`: キー付きサブファイル

### 3.2 データベースファイルの定義
```rpg
     FTOKMSP  IF  E           K        DISK
```

- `IF`: 入力ファイル（Input File）
- `K`: キー付きファイル
- `DISK`: ディスク装置

## 4. サブルーチン

### 4.1 サブルーチン名の制限
- **特殊文字（¥、$など）は使用不可**
- 英数字のみ使用可能
- 例: `¥CLEAR` → `CLEAR` に変更

### 4.2 初期化サブルーチンの例
```rpg
     C           CLEAR     BEGSR
     C                     MOVE *BLANK    TKBANG
     C                     MOVE *BLANK    TKNAKN
     C                     MOVE *BLANK    TKADR1
     C                     MOVE *BLANK    TKADR2
     C                     MOVE *BLANK    TKPOST
     C                     MOVE *BLANK    TKTELE
     C                     Z-ADD0         TKGEND
     C                     Z-ADD0         TKUZAN
     C                     ENDSR
```

## 5. 検索ロジック

### 5.1 キー付きファイルの検索
```rpg
     C                     MOVEL*BLANK    TKBANG
     C                     MOVELTOKTAN    TKBANG
     C           TKBANG    SETLLTOKMSP               30
```

**ポイント**:
- `MOVEL`: 左詰めで移動
- `SETLL`: キー値以上の最初のレコードを位置付け
- インジケーター30: レコードが見つからない場合にON

### 5.2 ファイルの選択
- **物理ファイル（TOKMSP）**: 主キー（TKBANG）で検索
- **論理ファイル（TOKMSL02）**: 代替キー（TKNAKN）で検索

**検索フィールドとファイルの対応**:
- 得意先番号（TKBANG）で検索 → TOKMSP（物理ファイル）
- 得意先仮名（TKNAKN）で検索 → TOKMSL02（論理ファイル）

## 6. 画面ファイル（DDS）

### 6.1 サブファイルの構造
```dds
     A          R PANEL80                   SFL
     A            TKBANG         5     10  4
     A            TKNAKN        20     10 11
     A            TKADR1        20     10 32
     A            TKPOST         6     11 10
     A            TKTELE        13     11 17
     A            TKADR2        20     11 32
     A            TKGEND         9  0  10 55EDTCDE(J)
     A            TKUZAN         9  0  10 68EDTCDE(J)
```

**フィールド定義の形式**:
- フィールド名、長さ、行位置、列位置の順
- `EDTCDE(J)`: カンマ編集（数値をカンマ区切りで表示）

### 6.2 サブファイル制御レコード
```dds
     A          R PANEL90                   SFLCTL(PANEL80)
     A                                      SFLSIZ(20) SFLPAG(7)
     A                                      OVERLAY PROTECT
     A                                      CA03(03 '終了')
     A                                      CF08(08 '次のレコード')
     A  41                                  SFLDSPCTL SFLDSP SFLEND
     A  42                                  SFLCLR
```

**重要なキーワード**:
- `SFLCTL`: サブファイル制御レコード
- `SFLSIZ`: サブファイルのサイズ
- `SFLPAG`: 1ページあたりの行数
- `SFLDSPCTL`: サブファイル制御レコードを表示
- `SFLDSP`: サブファイルを表示
- `SFLCLR`: サブファイルをクリア

### 6.3 見出し行の配置
```dds
     A                                  7  3'得番'
     A                                  7 11'得意先名'
     A                                  7 32'住所１／２'
     A                                  7 57'信用限度'
     A                                  7 72'売掛残'
     A                                  8  8'郵便番号'
     A                                  8 20'電話番号'
```

**配置のポイント**:
- 見出しはデータフィールドの開始位置より少し左に配置
- 複数行の見出しを使用して、レイアウトを整理

## 7. サブファイルの行位置

### 7.1 ヘッダーとデータの配置
- **ヘッダー行**: 通常7-8行目
- **データ行**: 10行目以降（ヘッダーとの間に1行空ける）

**理由**: ヘッダーとデータが重ならないようにするため

### 7.2 CPD7812エラーの回避
```
CPD7812: サブファイル制御レコードとサブファイルレコードが重複
```

このエラーを避けるため：
- サブファイル制御レコード（PANEL90）のヘッダー: 7-8行目
- サブファイルデータレコード（PANEL80）: 10行目以降

## 8. よくあるコンパイルエラーと対処法

### 8.1 QRG7064: ファイル名が長すぎる
**エラー**: ファイル名が8文字を超えている
**対処**: ファイル名を8文字以内に短縮

### 8.2 QRG5008/QRG5024: サブルーチン名が不正
**エラー**: 特殊文字（¥）を使用している
**対処**: 英数字のみのサブルーチン名に変更

### 8.3 QRG2001: H仕様書がない
**エラー**: ヘッダー仕様書が定義されていない
**対処**: H仕様書を追加

## 9. プログラム修正のベストプラクティス

### 9.1 元のファイルを保護
- 修正前に別名でコピー
- 例: `iph210.rpg` → `iph210l2.rpg`

### 9.2 段階的な修正
1. 画面ファイル（DDS）の修正
2. RPGプログラムのファイル定義修正
3. 初期化処理の修正
4. コンパイルとテスト

### 9.3 影響範囲の確認
- 画面レイアウトの変更
- データベースアクセスの変更
- 既存処理への影響

## 10. 参考プログラム

このプロジェクトの参考プログラム：
- `QEOL400/QEOLRPG/iph210.rpg`: 得意先照会プログラム
- `QEOL400/QEOLDSP/iph210s.dspf`: 得意先照会画面
- `QEOL400/QEOLRPG/iph120.rpg`: 受注入力プログラム（H仕様書の参考）

## まとめ

RPG III画面開発では、以下の点に注意してください：

1. ✅ ファイル名は8文字以内
2. ✅ H仕様書を必ず定義
3. ✅ サブルーチン名は英数字のみ
4. ✅ サブファイルのヘッダーとデータ行を分離
5. ✅ 検索フィールドに応じた適切なファイルを選択
6. ✅ 初期化処理で全フィールドをクリア

これらのルールに従うことで、エラーの少ない、保守性の高いRPG III対話型プログラムを作成できます。
