# LAB2: プログラムの修正 🔧

## 🎯 このラボの目標

IBM Bobを使って、既存の対話型プログラムに新しい項目を追加する方法を学びます。実務でよくある「機能追加」のシナリオを体験します。

**所要時間**: 10-15分  
**難易度**: ★★☆☆☆（初中級）  
**使用モード**: Code モード

---

## 📚 学習内容

このラボでは、以下のスキルを習得します：

- ✅ 修正箇所の特定方法
- ✅ 画面ファイル（DDS）の修正
- ✅ RPGプログラムの修正
- ✅ 影響範囲の分析
- ✅ 修正内容のレビュー

---

## 🚀 ステップ1: 事前準備（2分）

### 1.1 ファイルを開く

以下のファイルを開きます：

```
1_ibmi-hands-on/QEOLRPG/iph210.rpg      # RPGプログラム
1_ibmi-hands-on/QEOLDSP/iph210s.dspf    # 画面ファイル
1_ibmi-hands-on/QEOLDBF/tokmsp.pf       # 得意先マスター定義
```

### 1.2 現状確認

まず、プログラムの現在の状態を理解しましょう。

**Bobへの質問例**:
```
iph210.rpgとiph210s.dspfの関係を説明してください。
このプログラムは何をしていますか？
```

**期待される回答のポイント**:
- 対話型プログラムの構造
- 画面ファイルとRPGプログラムの連携
- サブファイルの役割

## 💡 ステップ2: 追加する項目の決定（3分）

### 2.1 利用可能な項目を確認

データベースにどんな項目があるか確認しましょう。

**Bobへの質問例**:
```
tokmsp.pfファイルに定義されている全項目を表形式で教えてください。
現在iph210s.dspfで表示されていない項目はどれですか？
```

**期待される回答のポイント**:
- 得意先マスターの全項目リスト
- 現在表示されている項目
- 追加可能な項目の候補

### 2.2 追加項目の選定

**今回追加する項目**:
1. **郵便番号（TKPOST）** - 6桁
2. **電話番号（TKTELE）** - 13桁

**Bobへの質問例**:
```
iph210の画面に郵便番号（TKPOST）と電話番号（TKTELE）を追加したいです。
現在の画面レイアウトを考慮して、どこに配置するのが適切か提案してください。
```

💡 **ヒント**: Bobは既存のレイアウトを分析して、適切な配置を提案してくれます。

## 🔍 ステップ3: 画面ファイルの修正（5分）

### 3.1 現在のレイアウトを確認

**Bobへの質問例**:
```
iph210s.dspfのPANEL80レコード（サブファイル明細）の
現在のレイアウトを図で説明してください。
```

**期待される回答**:
```
現在の表示項目：
行8: 得意先番号(4桁) 得意先名(11桁) 住所1(32桁) 信用限度額(55桁) 売掛残高(68桁)
行9: 住所2(32桁)
```

### 3.2 修正案の作成

**Bobへの質問例**:
```
PANEL80レコードに以下の項目を追加してください：
- 郵便番号（TKPOST）: 6桁、行8の10桁目から表示
- 電話番号（TKTELE）: 13桁、行9の10桁目から表示

修正後のDDSコードを提示してください。
```

**期待される回答例**:
```dds
     A          R PANEL80                   SFL
     A            TKBANG         5      8  4
     A            TKPOST         6      8 10
     A            TKNAKN        20      8 17
     A            TKADR1        20      8 38
     A            TKADR2        20      9 38
     A            TKTELE        13      9 10
     A            TKGEND         9  0   8 61EDTCDE(J)
     A            TKUZAN         9  0   8 74EDTCDE(J)
```

💡 **ヒント**: 生成されたコードをレビューして、桁位置が適切か確認しましょう。

### 3.3 見出し行の修正

**Bobへの質問例**:
```
PANEL90レコードの見出し行も、追加した項目に合わせて修正してください。
```

**期待される回答例**:
```dds
     A                                  7  3'得番'
     A                                  7 10'郵便番号'
     A                                  7 17'得意先名'
     A                                  7 38'住所１／２'
     A                                  8 10'電話番号'
     A                                  7 63'信用限度'
     A                                  7 78'売掛残'
```

## 🛠️ ステップ4: RPGプログラムの修正（3分）

### 4.1 必要な修正箇所の確認

**Bobへの質問例**:
```
iph210.rpgで、画面に追加した項目（TKPOSTとTKTELE）を
表示するために必要な修正箇所を教えてください。
```

**期待される回答のポイント**:
- サブファイルへの書き出し処理（WRITE PANEL80）
- クリア処理（¥CLEARサブルーチン）
- 特に修正不要な箇所の説明

### 4.2 クリア処理の修正

**Bobへの質問例**:
```
48行目からの¥CLEARサブルーチンに、
TKPOSTとTKTELEのクリア処理を追加してください。
```

**期待される回答例**:
```rpg
C           ¥CLEAR    BEGSR
C                     MOVE *BLANK    TKBANG
C                     MOVE *BLANK    TKNAKN
C                     MOVE *BLANK    TKADR1
C                     MOVE *BLANK    TKADR2
C                     MOVE *BLANK    TKPOST
C                     MOVE *BLANK    TKTELE
C                     Z-ADD0         TKGEND
C                     Z-ADD0         TKUZAN
C                     ENDSR
```

### 4.3 修正内容の確認

**Bobへの質問例**:
```
今回の修正で、プログラムの動作に影響はありますか？
テストで確認すべきポイントを教えてください。
```

**期待される回答のポイント**:
- データベースから自動的に項目が読み込まれる
- 画面表示の確認が必要
- レイアウトの崩れがないか確認

## 📋 ステップ5: 修正内容のレビュー（2分）

### 5.1 変更点のサマリー

**Bobへの質問例**:
```
今回の修正内容を以下の形式でまとめてください：
1. 修正したファイル
2. 追加した項目
3. 変更した行数
4. 影響範囲
```

**期待される回答例**:
```
修正サマリー：

1. 修正ファイル
   - iph210s.dspf (画面ファイル)
   - iph210.rpg (RPGプログラム)

2. 追加項目
   - 郵便番号（TKPOST）
   - 電話番号（TKTELE）

3. 変更箇所
   - PANEL80レコード: 2行追加
   - PANEL90レコード: 2行追加
   - ¥CLEARサブルーチン: 2行追加

4. 影響範囲
   - 画面レイアウトの変更
   - 表示項目の増加
   - プログラムロジックへの影響なし
```

## 💡 このラボで学んだこと

### IBM Bobでできたこと

✅ **修正箇所の特定が簡単**
- 従来: 仕様書を読んで、関連ファイルを全て調査（数時間）
- Bob活用: 「どこを修正すべきか」を質問して即座に回答（数分）

✅ **コード生成が自動化**
- 追加する項目を指定するだけで、DDSコードを生成
- 既存のコーディング規約に従った提案

✅ **影響範囲の分析が容易**
- 修正による影響を事前に確認
- テストポイントの洗い出し

### 実務での活用シーン

1. **機能追加の見積もり**
   - Bobに修正箇所を聞いて、工数を算出
   - 影響範囲を分析して、リスクを評価

2. **レビュー作業**
   - Bobに修正内容をサマリーさせて、レビュー資料を作成
   - 変更点を明確にして、承認プロセスを効率化

3. **テスト項目の作成**
   - Bobに影響範囲を聞いて、テストケースを作成
   - 網羅的なテスト計画を立案

## 🎯 よくある質問

**Q1: 画面レイアウトが崩れた場合は？**

A: Bobに「レイアウトを調整してください」と再依頼しましょう。桁位置を具体的に指定して修正を依頼すると、より正確な結果が得られます。

**Q2: 既存の項目を移動したい場合は？**

A: 「TKBANGを5桁目から10桁目に移動してください」と具体的に指示しましょう。

**Q3: 複数の修正を一度に依頼できる？**

A: 可能ですが、段階的に進める方が確実です。1つずつ確認しながら進めることを推奨します。

```
[ファイル名]に以下の項目を追加してください：
- 項目名: [フィールド名]
- 桁数: [長さ]
- 表示位置: 行[X]の[Y]桁目
- 編集コード: [編集指定]
```

### レイアウト変更
```
[レコード名]のレイアウトを以下のように変更してください：
- [項目名]を[位置]に移動
- [項目名]を削除
- 見出しも合わせて修正
```

### コードレビュー
```
以下の修正内容をレビューしてください：
[修正したコード]

確認ポイント：
- 構文エラーはないか
- 既存の処理への影響はないか
- パフォーマンスへの影響はないか
```
